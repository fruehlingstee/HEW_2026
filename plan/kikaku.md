# 企画書:セミオーダーケーキ プラットフォーム(仮)

作成日:2026-07-17 / ステータス:仮決定(ブレストからの収束案)

## 1. 概要

サイズ・スポンジ・クリーム・フルーツなどを既定パーツから選び、**飾り付けはユーザー自身が3Dプレビュー上で自由に配置**できるセミオーダーケーキ注文サービス。

注文データから店舗向けの**製造指示書(飾り配置図つき)を自動生成**し、予約枠・決済・受け渡しまでを一気通貫で扱う。ケーキ店に導入してもらう**SaaS(プラットフォーム提供型)** として設計し、HEWでは架空の1店舗でデモする。

## 2. 背景・課題

- オーダーケーキは現状、電話・店頭・LINEでの曖昧なやり取り(「こんな感じで」と画像を送る)が主流。完成イメージの齟齬、聞き漏らし、価格の不透明さがトラブルの元
- 店側は注文の聞き取り・見積もり・仕様の記録に時間を取られ、小規模店ほど負担が大きい
- 買い手は「自分でデザインした」体験を求めるが、フルオーダーは高価で敷居が高い

**→ セミオーダー(既定パーツ)+ユーザー配置(自由な飾り付け)の中間形で、双方の課題を解決する。**

## 3. ターゲット

| 区分 | 対象 |
|---|---|
| 買い手 | 誕生日・記念日にケーキを贈る個人(メイン:10〜30代)、推し活のオタクケーキ需要 |
| 導入側 | 個人経営〜中規模のケーキ店(オーダー対応の工数を減らしたい店) |

## 4. 提供価値

- **買い手**:完成形を3Dで確認してから注文できる/「自分で飾り付けた」体験/価格がリアルタイムに見える/アレルゲンが自動表示される
- **店舗**:聞き取り工数ゼロで仕様が確定/配置図つき製造指示書が自動で出る/予約枠と決済の管理が一体化

## 5. 主要機能

### 買い手側

1. **ベース選択(セミオーダー)**:サイズ→スポンジ→クリーム→外周デコをステップ形式で選択。選ぶたびに価格が即時更新
2. **飾り付けエディタ(ユーザー製作)**:3Dケーキにフルーツ・チョコ・飾りをドラッグ&ドロップで自由配置。回転・俯瞰で確認
3. **メッセージプレート**:手書き入力(Canvas)または書体選択
4. **アレルゲン自動表示**:選択パーツから自動集計
5. **受け取り**:店頭受取(受取番号)/配送(日時指定・冷蔵配送料計算)
6. **決済**:事前決済(Stripeテストモード)。注文確認・進捗通知(受付→製造中→受取可)
7. **キャンセル・返金**:規定(例:前日まで100%、当日50%)に基づく自動返金処理

### 店舗側

1. **パーツCMS**:パーツ(スポンジ・フルーツ・飾り)の登録・価格・在庫・アレルゲン・提供可否の管理
2. **予約枠管理**:1日の製造キャパ設定、枠の埋まり状況表示
3. **注文管理**:注文一覧、ステータス更新(受付→製造中→受取可→完了)
4. **製造指示書の自動生成**:構成一覧+飾り配置図(3D配置データから俯瞰図を生成)+メッセージプレート原稿

## 6. 技術構成

| 領域 | 技術 | 見せ場 |
|---|---|---|
| 3Dプレビュー | Blender(モデリング)→ glTF → Three.js(React Three Fiber) | レイキャストによる表面へのドラッグ配置。ローポリ+Draco圧縮でモバイル対応 |
| フロントエンド | React | ステップ式UIと3Dエディタの状態管理 |
| バックエンド | 任意(チームで決定) | 見積もり計算、予約枠、注文ステート管理 |
| DB | RDB(必須要件) | 注文・決済・返金の全履歴、パーツ構成、配置座標(JSON) |
| 決済 | Stripe テストモード | 決済・返金のトランザクション |

**配置座標データはDBに保存し、プレビューと製造指示書の両方で共用する**(1つのデータから2つのアウトプット)。

## 7. お金の流れ(IH専攻規定への適合)

- 注文時:事前決済 → 決済履歴をDBに保存
- キャンセル時:規定に基づく返金額を自動計算 → 返金トランザクションを記録
- 配送時:冷蔵配送料を距離・日時から算出し合算
- SaaSとして:店舗から月額利用料+決済手数料を徴収するモデル(ビジネス性の説明用)
- **全ての金銭のやり取りをデータとして残す(規定準拠)。実際の決済はStripeテストモードで想定**

## 8. HEW展示戦略

- **短い滞在時間対応**:「飾り付け→見積もり表示」を1〜2分で完結する体験デモに設計。会場用に選択肢を絞ったデモモードを用意
- **触って楽しい**:3Dドラッグ配置は例年人気の「体験できるもの」に合致。完成ケーキのスクリーンショットを持ち帰れる(SNS導線)
- **ゲーム・CG学科の票**:Three.jsの3D操作+Blenderモデリングは両学科の目を引く
- **審査員(教官・講師)**:製造指示書の自動生成・返金トランザクション・マルチテナント設計という「実務の裏側」で技術力を示す。ケーキ題材は例年のHEWに少なく「珍しいもの」評価を狙える

## 10. 体制(5人・案)

- 3D担当(Three.js/R3F)×2:飾り付けエディタ専任(最重要リスクのため固定アサイン)
- バックエンド×2:注文・決済・返金・予約枠・指示書生成
- フロントエンド(2D)+Blenderモデリング×1:ステップUI、店舗管理画面、パーツモデル制作

## 11. リスクと対策

| リスク | 対策 |
|---|---|
| Three.js習得が想定より重い | 9〜10月にPoCで検証。ダメなら2.5D(俯瞰配置図+側面プレビュー)へ縮退。俯瞰配置図は製造指示書と共用できるため無駄にならない |
| 3Dエディタに工数が偏り決済系が薄くなる | Ver0.5時点で決済・注文管理を先に完成させる(スケジュール参照) |
| 「ただのECでは」と評価される | 製造指示書生成・返金処理・マルチテナント設計を発表で前面に出す |
| デモ渋滞(1人の体験が長い) | デモモードで選択肢を絞る+タブレット複数台 |

## 12. 自己評価(ブレスト時の3指標)
