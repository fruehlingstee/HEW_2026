# 企画書：パートナーショッピングEC（仮）

## 0. 一言コンセプト
「パートナーと一緒に服を選んでいるような感覚で買い物ができる」リアル調AI試着×パートナーシミュレーターのアパレルECサイト。
テーマ①（3D試着アパレルEC）とテーマ③（パートナーシミュレーター）を掛け合わせ、"疑似パートナーと過ごすショッピングデート"を体験させる。
※試着表現は事前生成したリアル調静止画方式を採用（詳細は5章・8章）。テーマ①の名称は「3D試着」だが、応募要件が技術的に3D実装を必須としているかは別途確認が必要。

## 1. 背景・狙い
- HEWテーマは「斬新で画期的で今までにない『お金』が動くWebサービス」。単なるECだと類似作品が多く埋もれやすいため、パートナーシミュレーター要素を掛け合わせて差別化する。
- 審査員（教官・講師）は毎年多くの作品を見ているため「珍しさ」が評価されやすい → キャラクターとの関係性×購買という組み合わせは珍しい。
- 学生投票はHALの学科構成上、ゲーム学科・CG学科の比率が高い → キャラクターデザインや関係性システムのような「ゲーム的・体験的」要素は共感/得票されやすい。
- 例年「体験できるもの」「触って楽しいもの」が人気 → リアル調試着ビジュアルの操作（モデル選択・コーデ切り替え）、キャラクターとの掛け合いはこの傾向に合致。
- 審査は短時間で行われる（見学時間が短い）ため、最初の数十秒で「これは何のアプリか」「何が楽しいか」を伝えられる導線が必須。
- 過去の金賞・銀賞作品を確認した結果、審査員は「珍しさ」だけでなく完成度・実装の裏付けも見ていると考えられる。銀賞作品はUnity WebGL＋SNS＋DM等、技術要素を広げすぎたことでコア機能（購入者限定ダウンロード制御）を提出直前にデモ用へ一時的に無効化する状態で提出されており、野心的すぎるスコープは未完成リスクに直結する。本企画ではこの教訓を踏まえ、機能を段階的にリリースし核機能を最優先で固める方針とする（詳細は5章・10章）。

## 2. コンセプトストーリー
ユーザーは「バーチャルパートナー」とアプリ内でショッピングデートをする。
パートナーが服をおすすめしたり、試着した姿にその場でコメント（ボイス付き）したり、一緒に買い物カゴを選んだりする。
好感度のような蓄積型パラメータは持たず、購入・試着・会話などその場のアクションに対してパートナーがリアルタイムに反応する設計とする（金賞・銀賞作品にもこの種の蓄積パラメータ設計はなく、実装・調整コストを抑えつつ短時間デモでも伝わりやすい設計として妥当と判断）。

## 3. ターゲット層
- メイン：ファッションEC＋育成/関係性ゲームに関心がある層（10〜20代）
- HEW文脈でのターゲット：審査員（新規性重視）／学生投票者（1・3年生、ゲーム・CG学科多め）

## 4. お金が動く仕組み（レギュレーション対応）
必須：Webアプリ、フロント/バックエンド分離、DB必須、金銭のやり取り履歴をデータとして保持（実際の決済は行わずStripeテストモード等で想定）。

**採用方針（確定）：金銭の動きはまず「EC購入」を核として確定させる。フリマ（ユーザー間取引）は開発に余力があれば後日追加するストレッチゴールとし、v0.1〜v1.5のコアスコープからは一旦外す（詳細は5章v2.0参照）。**
| 種別 | 内容 | 状態 |
|---|---|---|
| EC（購入） | リアル調試着ビジュアルで確認した服・アイテムの購入（ショップからの新品購入） | **確定（核機能）** |
| フリマ（ユーザー間取引） | ユーザーが着なくなった服を出品し、他の実ユーザーが購入する。出品はNPC/ダミーデータを使わず実ユーザーのみ | **保留（余力があれば追加）** |
| 投げ銭/ギフト | パートナーへのプレゼント購入 | 不採用 |
| 共同ウォレット/割り勘購入 | 二人の財布を貯めて共同購入 | 不採用 |
| メンバーシップ/サブスク | 月額会員プラン | 不採用 |
| デート/コーデ回数券 | チケット制課金 | 不採用 |
| 記念日サプライズ課金 | 記念日課金 | 不採用 |
| ガチャ的要素 | ランダム要素の「福袋」等 | 不採用（ギャンブル系NG規定のリスク） |

採用理由：EC購入は「お金が動くWebサービス」規定を満たす最小かつ確実な導線であり、実装リスクが低い。フリマは実ユーザー間で双方向にお金が動く点で差別化効果が高く、HEW資料に例示されている「フリーマーケットなどの物の取引」にも直接合致するが、共通ウォレット連携まで含めると実装難度が高く、金賞・銀賞事例で見られた「野心的すぎるスコープによる核機能未完成」のリスクを避けるため、開発初期のコアスコープからは外す。フリマに充てる予定だった開発リソースはリアル調試着ビジュアルの作り込みに再配分し、開発に余力が生まれた段階でフリマを追加する。

→ EC購入の履歴をDBに保存し「お金の動き」を証跡として残す。

**財布の連携（保留）**：フリマを追加する場合は、EC購入とフリマで共通の財布（残高）を使う設計とし、フリマで服を売って得たお金をそのままEC購入時の支払い・値引きに充当できるようにする。「売って、稼いで、また買う」という循環は差別化の目玉になり得るが、v0.1〜v1.5では実装しない。

**開発方針（確定）**：v0.1〜v1.5はEC購入・リアル調試着・パートナー演出を核として完成度を優先する。フリマ（共通ウォレット連携含む）は最も実装難度が高い部分であるため、コア機能が安定してから着手するストレッチゴール（v2.0）として扱う。

**運用方針（保留）**：フリマを実装する場合、出品はNPCを使わず実ユーザーのみとし、展示当日に商品が並んでいる状態にするため、チームメンバーが自分たちの実アカウントで事前に出品しておく運用とする。

## 5. コア機能案（段階リリース計画）
差別化の核を「リアル調試着×パートナー演出の作り込み」に置き、EC購入導線を土台として最優先で完成させる。フリマは開発に余力があれば追加するストレッチゴール（v2.0）とする。

### v0.1（MVP・最優先）：お金が動く基本導線
- ユーザー認証（メール＋パスワード、Supabase Auth）
- 商品一覧・カート・購入導線（EC標準機能、ショップからの新品購入）
- マイページ：購入履歴のログ表示（レギュレーションのデータ保持要件を満たす）
- ※フリマ・共通ウォレット機能はv0.1のスコープから除外（v2.0で検討、4章参照）

### v0.5：リアル調試着ビジュアルシステム（開発の主軸として強化）
- 選択式の固定モデル（3〜5体程度）を用意し、ユーザーはモデルを選んで試着姿を確認する（ユーザー自身のアバター化は不採用。個人ごとの試着生成はリアルタイム生成が前提になり実装難度・インフラコストが跳ね上がるため）
- 各モデル×コーディネートの組み合わせは、画像生成AIツールで展示前に事前生成し静止画として用意する（本番アプリ側でのリアルタイム生成はしない）
- 360度回転・ズームは行わず、複数角度（正面・斜め・背面等）の事前生成画像を切り替える疑似回転演出で代替
- フリマに充てる予定だった工数をこちらに再配分し、生成画像のクオリティ（質感・光沢の再現度）とコーデバリエーション数を作り込む
- レイヤー着用（重ね着）は動的合成ではなく、レイヤー込みの完成コーデとして事前生成する

### v1.0：パートナーシステムと連動機能
- パートナーキャラクターは2タイプ（男女1体ずつ）を用意し、ユーザーが選べる形式で採用（工数を抑えつつ選択の幅を確保）
- キャラクター方針：オリジナルキャラクターで採用（VOCALOID系はライセンス確認の手間を考慮し保留）
- 好感度のような蓄積パラメータは不採用。関係性は数値化せず、その場のリアクションで表現
- 試着中のリアクション（褒める/困った顔をする等）をボイス付きでキャラクターが返す
- パートナーにも服を着せ替えられ、パートナー自身が試着した際は恥ずかしがる・気に入る等の専用リアクション（ボイスあり）を用意し「一緒に着替える」体験を強化（パートナーの全コーデ分の画像もv0.5と同様に事前生成する）
- 「パートナーと一緒にコーディネートを選ぶ」画面（ユーザーが選んだモデルとパートナー、2人分の静止画を並べて表示、ペアルック/カップルコーデ提案）
- ショッピング：「一緒に選ぶ」レコメンド（パートナーが似合う服を提案）
- フリマ：パートナーが「これ出品しちゃう？」「掘り出し物ないか一緒に見てみよう」と絡んでくる演出

### v1.5：仕上げ・展示対応
- 展示当日は、来場者（審査員・学生）もその場でアカウントを作成し、購入体験ができるようにする
- UX/UI仕上げ、デモ導線の磨き込み

### v2.0（余力があれば・ストレッチゴール）：フリマ機能の追加
- フリマ機能：出品（着なくなった服を自分のクローゼットから出品。価格は出品者が自由に設定、実ユーザーのアカウントに紐づく）
- フリマ機能：購入（他の実ユーザーが出品した商品一覧を閲覧・購入。購入するとお金が出品者に、商品が購入者に移動）
- EC購入とフリマ共通の財布（残高）連携：フリマの売上はそのままショップでの新品購入に使える
- NPC/ダミー出品は使わない。展示当日までにチームメンバーが実アカウントで事前出品しておく運用
- パートナーが「これ出品しちゃう？」「掘り出し物ないか一緒に見てみよう」と絡んでくる演出（v1.0のパートナー機能と連動）

## 6. 差別化ポイント（審査員・学生投票向け）
- EC単体でも恋愛ゲーム単体でもない掛け合わせ企画で「珍しさ」を担保
- フリマに充てる予定だった開発リソースをリアル調試着ビジュアルの作り込み（生成画像のクオリティ・コーデバリエーション等）に集中させ、ビジュアル訴求力をさらに高める
- リアル調AI生成のビジュアルは写真同等のクオリティで映えるため、短時間デモでも魅力が伝わりやすい
- キャラクター性・関係性システムはゲーム/CG学科の学生の興味・投票と親和性が高い
- ITの専門的な仕組み（DB設計や決済フロー）は裏側に隠しつつ、表側は「体験・触って楽しい」を前面に出せる

## 7. デモ・展示戦略（短時間審査対策）
- 冒頭10〜20秒で「パートナーと一緒に試着してショッピングできるアプリ」と一目で伝わるビジュアル（リアル調キャラ+試着画面）を見せる
- その場で操作できる試着体験（モデル選択・コーデ切り替え）を用意し「触れる」楽しさを重視
- パートナーがボイス付きでリアルタイムに反応する瞬間（褒めてくれる、驚くなど）をデモの山場にする

## 8. 技術・データ設計

### 8-1. 技術スタック（優先度付き）
5人チーム、言語・技術を極力絞った構成を提案。**デプロイする方針が確定したため、本番環境で永続化できる構成に変更。** v0.1のMVPはNext.js／Supabase／Prisma／Stripeのみで完成させ、試着ビジュアル・ボイス関連はv0.5以降で導入することで、銀賞事例（Unity＋MediaPipe等マルチスタックによる実装遅延）のようなリスクを抑える。フリマ機能は保留（4章参照）とし、浮いた開発リソースをv0.5以降のリアル調試着ビジュアル制作に重点配分する。

| 領域 | 技術 | 理由 | 優先度 |
|---|---|---|---|
| フロント＋バックエンド | Next.js（TypeScript/JavaScript） | 1プロジェクトでフロントとAPIの両方を書け、学習コストと構成の手間を削減。情報量が多くAIの支援も受けやすい | Must（v0.1） |
| ホスティング/デプロイ | Vercel | Next.jsとの親和性が高く、GitHub連携で自動デプロイできる。無料枠で展示用途に十分 | Must（v0.1） |
| DB | Supabase（PostgreSQL） | デプロイ環境（サーバーレス）では前提のSQLiteは永続化に不向きなため変更。無料枠あり | Must（v0.1） |
| ORM | Prisma | SupabaseのPostgreSQLに対してPrismaでスキーマ管理・クエリを行う。学習コストが低く定番の組み合わせ | Must（v0.1） |
| ユーザー認証 | Supabase Auth | メール＋パスワード認証を自前実装せず利用可能。来場者のその場アカウント作成にも対応しやすい | Must（v0.1） |
| 決済シミュレーション | Stripe（テストモード） | レギュレーションで例示済み、ドキュメントが豊富 | Must（v0.1）※EC・フリマの金銭導線そのもの |
| 構成管理 | GitHub ＋ Vercel自動デプロイ＋ Notion/Trello等でタスク管理 | 小人数〜中人数チームの定番、pushで自動デプロイされ確認が楽 | Must（v0.1から運用） |
| スタイリング | Tailwind CSS | デザインスキルが薄くても整った画面を組みやすい | Must（全フェーズ） |
| 試着ビジュアル表示 | 通常の画像コンポーネント（Next.js Image等） | 事前生成した静止画を表示するだけのため3D描画ライブラリは不要。実装コストを大幅に削減できる | Must（v0.5） |
| 試着ビジュアル制作ツール（開発環境のみ・本番スタック外） | 画像生成AIツール/API（Stable Diffusion系または商用バーチャル試着サービス等、選定は10章ToDo参照） | 展示前にモデル×コーデの組み合わせを事前生成するための制作用ツール。利用規約上の商用利用可否・生成物の権利関係を要確認 | v0.5着手 |
| ファイルストレージ | Supabase Storage | 事前生成した静止画・ボイス音源などの静的ファイルの配置先 | v0.5以降Must |
| ボイス（音声演出） | VOICEVOX（無料の日本語音声合成エンジン） | 声優を確保せずにキャラクターボイスを実現できる。セリフのテキストから音声を生成する方式。声優収録に切り替えることも可能 | Nice-to-have（v1.0） |

### 8-3. データ設計
- バックエンド：ユーザー/商品/注文API（フリマ・好感度ロジックは無し。フリマは保留、追加時は下記拡張候補テーブルを実装）
- DB主要テーブル（案）：users, partners（キャラ設定）, items（ショップ商品）, orders, order_items
- 試着ビジュアル管理：look_images（モデル×コーデの事前生成画像。model_id, item_combo_id, image_url, angle等）を新規追加
- 拡張候補（v2.0でフリマ追加時）：listings（フリマ出品：出品者user_id・価格・状態）, marketplace_transactions（フリマ取引履歴：出品者/購入者/金額/日時）
- ユーザー認証：**通常のメール＋パスワード認証で実装（確定）**。Supabase Authを利用し、来場者がその場でアカウント作成できるようにする想定
- 決済：Stripeテストモード等で疑似決済、実売上は発生させない
- 音声：**ボイス演出は採用確定**。VOICEVOX等のAI音声合成での実現を提案（8-1参照）
